# Database Schema

## Conventions

- Primary keys use UUIDs generated by PostgreSQL.
- Timestamps use `timestamptz` and are stored in UTC.
- The UI interprets and displays event times in `Asia/Kolkata`.
- `created_at` and `updated_at` default to `now()`; an update trigger maintains
  `updated_at`.
- SQL migrations are authoritative. Generated TypeScript types mirror them.

## Enums

### `event_theme`

- `ivory`
- `midnight`
- `botanical`

### `rsvp_status`

- `pending`
- `accepted`
- `declined`

## `events`

| Column            | Type          | Rules                                         |
| ----------------- | ------------- | --------------------------------------------- |
| `id`              | `uuid`        | Primary key, generated                        |
| `title`           | `text`        | Required, 1–120 characters                    |
| `subtitle`        | `text`        | Optional, at most 180 characters              |
| `event_date`      | `timestamptz` | Required                                      |
| `venue`           | `text`        | Required, 1–160 characters                    |
| `address`         | `text`        | Required, 1–500 characters                    |
| `google_maps_url` | `text`        | Optional HTTPS Google Maps URL, at most 2,048 |
| `theme`           | `event_theme` | Required, defaults to `ivory`                 |
| `created_at`      | `timestamptz` | Required, defaults to current time            |
| `updated_at`      | `timestamptz` | Required, defaults to and tracks current time |

Events are ordered by `event_date`, then `created_at`. Deletion uses
`ON DELETE RESTRICT`; an event with invitations must not be deleted.

## `invitations`

| Column           | Type          | Rules                                         |
| ---------------- | ------------- | --------------------------------------------- |
| `id`             | `uuid`        | Primary key, generated                        |
| `event_id`       | `uuid`        | Required foreign key to `events.id`           |
| `slug`           | `varchar(12)` | Required, unique, base64url characters only   |
| `guest_name`     | `text`        | Required, 1–120 characters                    |
| `custom_message` | `text`        | Optional, at most 1,000 characters            |
| `rsvp_status`    | `rsvp_status` | Required, defaults to `pending`               |
| `created_at`     | `timestamptz` | Required, defaults to current time            |
| `updated_at`     | `timestamptz` | Required, defaults to and tracks current time |

Slugs contain 72 bits of entropy from nine cryptographically random bytes
encoded as 12 base64url characters. Creation retries a unique collision a small,
bounded number of times.

## Indexes

- Unique index on `invitations.slug`.
- Index on `invitations.event_id` for joins and delete checks.
- Composite index on `(event_id, rsvp_status)` for dashboard filtering.
- Index on `events.event_date` for ordered event lists.
- Search initially uses bounded `ILIKE` queries; trigram indexes are deferred
  until data volume demonstrates a need.

## RSVP consistency

The only public state transition is:

```text
pending -> accepted
pending -> declined
```

The repository performs a conditional update matching both the invitation slug
and `rsvp_status = 'pending'`. PostgreSQL's update semantics make concurrent
submissions safe: one request changes the row and subsequent requests match no
pending row. Admin editing may reset or override status only if that capability
is deliberately added later.

## Row Level Security

RLS is enabled on both tables. No policies grant access to `anon` or
`authenticated`. Application reads and writes use a Supabase secret key from
server-only code, which bypasses RLS.

This prevents direct browser access but does not protect the intentionally
unauthenticated admin Server Actions from users who can access `/admin`.

## Validation split

PostgreSQL constraints enforce durable invariants such as required values,
lengths, enum membership, slug format, uniqueness, and relationships. Zod
schemas provide earlier, field-specific feedback. Both layers remain aligned;
client validation is an enhancement rather than a security boundary.
